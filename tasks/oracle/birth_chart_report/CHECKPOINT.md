## Current state summary

A first backend + prompt-context implementation exists for the Birth Chart Report feature, but the user explicitly wants the next iteration to focus on making the onboarding feel like the normal Oracle chat experience rather than a separate form or instant hardcoded bot dump.

The current implementation partially works:

- `users.birthChartReport` exists in Convex schema.
- `birth_chart_report_jobs` exists as an async generation queue.
- Report generation can be queued, processed, generated by an LLM, and saved to `users.birthChartReport.markdown`.
- The Oracle prompt pipeline can inject the completed report as canonical context.
- The `birth_chart` feature has been refactored toward mentor mode.
- A printable route exists at `/oracle/birth-chart-report`.

But the UX is not yet right:

- Hardcoded onboarding messages appear immediately, with no streaming/typing effect.
- Profiling questions are currently sent as separate assistant message bubbles.
- Answer options currently appear above the normal Oracle input and custom answers use the normal input bar, which is not desired.
- There is no polished indication in the active chat UI when report generation completes.
- The current questions are low leverage and produce weak profiling data.
- The generated report format is too vanilla Markdown and needs a richer content/design structure.

## Needed architecture change for questionnaire

The current backend stores answers step-by-step. The user now wants pagination and final review. Best next approach:

### Frontend state first

Hold questionnaire state locally inside chat page or a dedicated component, e.g.

```txt
src/components/oracle/birth-report/BirthReportQuestionnaire.tsx
```

State shape might be:

```ts
type BirthReportQuestion = {
  id: string;
  prompt: string;
  type: "single" | "multi" | "text" | "hybrid";
  options?: Array<{ label: string; value: string; description?: string }>;
  maxLength?: number;
};

type BirthReportQuestionnaireAnswers = Record<string, string | string[]>;
```

The component should:

- render one question at a time
- allow back/next
- support “skip”
- support custom text inside the module
- on final submit, call one mutation/action that saves all answers and queues generation

### Backend simplification

Add a mutation/action specifically for completed questionnaire submission, e.g.

```ts
api.birthChartReport.queue.submitReportQuestionnaire
```

It should:

1. validate/sanitize all answers
2. write them to `users.birthChartReport.profilingAnswers`
3. set status `pending` / onboardingStep `queued`
4. create queue job idempotently
5. schedule worker

This avoids using normal user messages as intermediate question answers.

### Chat messages

Avoid adding a user message for every questionnaire option. Instead, after final submit, optionally add one user message summarizing the completed questionnaire, e.g.

```txt
I answered the Birth Chart Report questions. Please create my report.
```

Or no user message, just assistant “I’m crafting your report now.” The user prefers it to feel conversational, but not cluttered.

## Hardcoded streaming animation requirement

Currently hardcoded onboarding assistant messages are inserted via `api.oracle.sessions.addMessage`, so they appear all at once.

Better approach options:

### Option A: Use existing streaming message storage

There are internal mutations in `convex/oracle/sessions.ts`:

- `createStreamingMessage`
- `updateStreamingContent`
- `finalizeStreamingMessage`

A new internal action could simulate streaming for hardcoded text:

```ts
internal.oracle.sessions.streamHardcodedAssistantMessage({
  sessionId,
  content,
  chunkDelayMs,
})
```

But Convex actions updating content many times may be expensive/noisy.

### Option B: Client-side fake streaming preferred

Recommended:

- Store hardcoded assistant message normally, but include metadata or detect `modelUsed === "birth_chart_report_onboarding"`.
- On the client, if a newly arrived assistant message has that model marker and has not been animated locally, render it through a fake typewriter component.
- Keep the DB content complete; animation is only presentation.

Pros:

- no Convex write loop
- no backend churn
- easy to tune for soothing effect

Implementation idea:

```tsx
function HardcodedStreamingMessage({ content }) {
  // reveal by words/chunks with requestAnimationFrame or interval
  // add subtle breathing cursor / shimmer
}
```

Use it only for onboarding hardcoded assistant messages.

Need to ensure old messages render normally when revisiting session.

## Questions are currently not good enough

The next session/agent should **not** blindly keep these fields/questions:

```ts
centralQuestion
publicPersona
innerExperience
pronouns
```

The current answers can be low-value:

```ts
centralQuestion: "All of those"
publicPersona: "I have no idea"
innerExperience: "Restless"
pronouns: "he/him"
```

The next agent’s task is to determine the highest-leverage questions for personalization.

### Goal for new questions

Questions should elicit information the birth chart alone cannot know, and that helps the report feel uncannily personal without becoming therapy intake.

They should help generate:

- report tone/language
- “central question” or life theme
- relationship to self/public image
- current season of life
- desired use of astrology
- mottos / affirmations / reminders
- strengths and shadows phrased in resonant language
- practical reflection prompts

### Candidate question categories to research/design

The next agent should decide the final set, but consider:

1. **Current life season**
   - “What chapter of life are you in right now?”
   - options like: rebuilding, choosing direction, healing, expanding, ending something, beginning something

2. **What they want the report to help with**
   - love/relationships, purpose/work, self-understanding, confidence, emotional patterns, timing/current transition

3. **Inner friction**
   - “Which pattern do you most want to understand?”
   - overthinking, people-pleasing, intensity, avoidance, restlessness, self-doubt, control, burnout

4. **How they want to be spoken to**
   - gentle, direct, poetic, practical, big-sibling, mystical, no-nonsense

5. **Optional identity/language**
   - pronouns
   - name/preferred name

6. **What they already know or believe about themselves**
   - allows the report to affirm/contrast with chart evidence

Avoid questions that produce vague answers like “All of those” unless multi-select is intentionally supported and stored structurally.

### Store structured answers

Rather than storing vague strings, store structured values:

```ts
profilingAnswers: {
  currentSeason?: string[];
  reportFocus?: string[];
  growthPattern?: string[];
  tonePreference?: string;
  preferredName?: string;
  pronouns?: string;
  customContext?: string;
}
```

This likely requires schema migration from current `profilingAnswers` shape.

Current report Markdown is basic headings and paragraphs. The user wants it richer and more useful.

Improve it to recognize/beautifully style:
